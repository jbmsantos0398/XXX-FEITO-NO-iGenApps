WCAG — A short guide to the rules & best practices

Short and simple explanations of WCAG rules. Use it as a quick guide to get started, but you may need to check more details for your own project requirements.

ÁUDIOS

Essential requirements

  • Keyboard-only navigation

    Everything on the page must work using only a keyboard. You should be able to move around with the Tab or Shift+Tab keys and choose things with the Enter key.

    <button>Open menu</button>
<a href="/next">Next step</a>
<input id="name">

    Note: Links use the Enter key. Buttons use Enter or Space. Radio buttons can be changed with the arrow keys.

  • No keyboard traps (focus should not get stuck)

    When a popup opens, it is okay to keep the focus locked inside, but you must also be able to leave it using just the keyboard.

    Make sure there is an easy way to close it, like the Esc key or a visible Close button you can reach by the keyboard. When it closes, focus should go back to the thing that opened it.

    // On modal close (example)
focusTrap.deactivate();
triggerButton.focus();

    Failure = user cannot leave the widget with the keyboard.

  • Meaningful alt text for images

    If an image adds important information, use an alt text that explains what the image shows. Screen readers read this text instead of the image.

    If an image is only for decoration and does not add meaning, then leave the alt text empty (alt="") so it is skipped.

    <img src="chart.png" alt="Sales rose 20% in Q2">
<img src="decorative.png" alt="">
  • Don’t use colour or visual clues on their own

    Do not use colour, shape, size, sound, or position by itself to explain something. People who cannot see colour or hear sounds still need to understand what to do. Always give information in another way too, such as text, labels, or icons with text.

    <!-- Bad example -->
<p>Fields marked in red are required.</p>;

    Tip: Colour and visual cues are fine to use, but they must not be the only way people get important information.

  • Make video and audio easy for everyone

    If a video has people talking, it must have captions. Captions (subtitles) show the spoken words as text on the screen.

    If important things happen on screen, explain them. This can be done with audio descriptions or a short written description.

    If something is audio-only, add a text transcript. If it is video-only, explain what it shows in text. 

    <video controls>
  <source src="talk.mp4">
  <track kind="captions" srclang="en" src="talk.vtt">
</video>

    Best practice Do not autoplay sound. Always give people simple controls to play, pause, or mute.

  • Avoid seizure-risk flashing

    Do not include flashing content that could trigger seizures. As a rule, avoid flashes that exceed the seizure thresholds.
    If in doubt, run a seizure-safety checker — flashes that are small, low contrast, and not mostly red are safer.

    General rule: Don’t use fast flashing. Avoid anything that flashes more than three times a second, unless it clearly meets the WCAG safety limits for size, contrast, and colour.

  • Logical reading & focus order

    Make sure elements appear in the HTML in the same order that people see them on the page. This keeps the reading and focus order predictable for screen-reader and keyboard users.

    Avoid shifting elements with CSS. What people see should match the order in the HTML.

  • Inputs must have labels or accessible names

    Every form control needs a descriptive label. Use <label> or a programmatic accessible name (for example aria-label) for icon-only controls. If visible text exists, make sure the accessible name includes that text so voice activation and screen readers match the visible label.

    <label for="email">Email</label>
<input id="email" … >

<button aria-label="Close">X</button>
  • Descriptive link & button text

    Link and button labels should clearly explain what they do. Avoid vague phrases like “Click here.” If the same label repeats on a page, ensure surrounding context makes each link’s purpose clear.

    <a href="/pricing">Compare pricing plans</a>
<button>Download report (PDF)</button>
  • Allow pausing or stopping of motion

    Provide controls to pause, stop, or hide moving/animated content that might distract or cause discomfort.

Level AA

Important improvements

  • Clear heading structure

    Use headings to form a logical outline (don’t skip levels without reason). This makes it easy for assistive tech users to scan and navigate.

    <h1>Page title</h1>
<h2>Section</h2>
<h3>Subsection</h3>

    Best practice: Use one main h1 heading to lead the page content.

  • Text contrast (minimum 4.5:1)

    Keep body text contrast at or above 4.5:1 against its background so people with low vision can read it.

  • Visible keyboard focus indicator

    Make focus visible with a clear outline or ring so keyboard users always know where they are on the page.

    :focus-visible {
  outline: 2px solid #06f;
  outline-offset: 2px;
}

    Tip: Make it a global feature where possible.

  • Text must work at 200% size

    Some people make text bigger so it is easier to read. Your page must still work when the text is twice as big (200%). At this size, words and buttons should not overlap, break, or become hard to use.

    A quick way to test this is to zoom in your browser:
    Windows: Press Ctrl + + a few times.
    Mac: Press Cmd + + a few times.

    Note: This is full-page zoom, not text-only resize, but it is usually enough to reveal layout issues. Text-only resizing may still require testing.

    Tip: Use pixel sizes for left and right spacing (margin or padding). It helps the layout stay readable when the text gets bigger.

  • Reflow to a single column on narrow viewports

    At narrow widths (320px or equivalent), most content should stack into a single column without horizontal scrolling. Large media like maps or wide tables may scroll horizontally but must still be usable.

  • Use comfortable touch targets

    Buttons and links should be big enough to tap easily, especially on touch screens. Touch targets should be at least 24px by 24px.

    Best practice: Larger targets are easier to use. Aim for around 44px by 44px wherever possible.

  • Use real text (not images of text) where possible

    Prefer HTML text so content can be resized, searched, translated, and read by assistive tech. If a design requires text as an image (for branding or decorative reasons), ensure an accessible text alternative is provided (e.g., visible text nearby or appropriate alt).

  • Consistent navigation

    Keep navigation components and their order consistent across pages so users don’t have to relearn the layout.

  • Consistent form labels & instructions

    Use the same patterns for labels, hints, and error messages across forms to reduce confusion.

    Generic best practice pattern (in error state):

    <label for="input_id">Label text</label>
<div id="input_hint_id">Hint text</label>
<-- Fill type, autocomplete and inputmode to suit: -->
<input id="input_id" aria-describedby="input_error_id input_hint_id" aria-invalid="true" type="" autocomplete="" inputmode="">
<div id="input_error_id">Error text and help</label>
  • Live captions for live audio/video

    Provide real-time captions for live streams (different from pre-recorded captions which are supplied as files).

  • Expose status updates to assistive tech

    Use live regions (for example role="status" or aria-live) to announce non-modal status messages without moving keyboard focus.

    <-- Preferred: -->
<div role="status">Saved</div>

<-- Alternative: -->
<div aria-live="polite">Saved</div>

    Best practice: Don't use live regions to report individual input errors.

Level AAA

Useful extras

  • Plain, easy language

    Use clear, simple wording and avoid jargon where possible. Add summaries or examples for complex content.

    Note: The average reading age in the UK is 9-11 years old, write accordingly.

  • Expand acronyms on first use

    Explain acronyms so everyone understands them.

    <p>Accessible Rich Internet Applications (ARIA)</p>
<abbr title="Accessible Rich Internet Applications">ARIA</abbr>
  • Higher contrast (7:1) where practical

    For people with very low vision, higher contrast improves readability — aim for 7:1 where practical.

  • No autoplay background audio

    Don’t start background audio automatically. If audio plays, provide a clear control to pause or stop it.

  • Extended audio descriptions

    For visually rich videos, provide longer audio descriptions so blind users can access important visual information.

  • Sign language options

    Consider offering sign-language interpretation for videos, where appropriate for your audience.

  • Help with pronunciation

    Offer phonetic hints for unusual names or terms when helpful (for example with an aria-describedby reference).

  • Allow re-authentication without data loss

    If a session times out, let users sign in again and come back to their unfinished work without losing input.

Best practices

Helpful tips

  • Warn when links open a new tab

    Tell users if a link opens a new window or tab to avoid surprises (for example add “(opens in new tab)” to the visible label).

    <a href="/docs" target="_blank" rel="noopener">Docs (opens in new tab)</a>

    Note: Use text if you can. If you use an icon, give it a title, alt text, or an accessible name.

  • Align accessible names with visible labels

    Make sure programmatic accessible names (ARIA) include the visible label text when present. This helps voice activation and prevents mismatches between what users see and what assistive tech reads.

    <button aria-label="Save changes">Save</button>
  • Provide a “Skip to main content” link

    Give keyboard users a way to jump past repeated navigation to the main content. Ensure the link is visible on focus.

    <a href="#h1" class="skip-link">Skip to main content</a>
...
<h1 id="h1" tabindex="-1">Main heading, which precedes the content</h1>

    Note: The tabindex="-1" lets skip-links send focus to the heading without adding it to the normal tab order.

    Best practice: Skip links cause fewer problems when they point to the main heading (h1) instead of the whole content area.

  • Prefer native HTML over ARIA

    Use built-in elements when possible. Only use ARIA when no native option exists.

    Remember: No ARIA is better than bad ARIA.

  • Clear inline help and examples

    Show short hints or examples near form fields so users know the expected format (for example date format).

    <label for="dob">Date of birth <small>(DD-MM-YYYY)</small></label>
  • Label icons used as buttons

    Icons used as buttons must have visible text or an accessible name (for example aria-label). If the control also shows text, include that text in the accessible name (label-in-name).

    <button aria-label="Play">▶</button>
  • Give helpful, actionable error messages

    Error messages should explain what went wrong and how to fix it. Place the message next to the field so users can spot the problem quickly, and make sure it is exposed programmatically.

    Connect the message to the input with aria-describedby and mark the field with aria-invalid="true" like so:

    <label for="email">Email address</label>
<input id="email" name="email" aria-describedby="email-error" aria-invalid="true">
<p id="email-error">Please enter a valid email, e.g. name@example.com.</p>

<!-- Use role="alert" only for submission-level messages -->
<div role="alert">Form submission failed — please fix the highlighted fields.</div>

    Best practice: Don't use role="alert" for each field error, as it can interrupt people, only use it upon form submission.

  • Use type, autocomplete and inputmode

    Help people fill in forms faster and with fewer mistakes. Use the right input type so the browser knows what the field is for.

    Use autocomplete so browsers can suggest saved details, like names or email addresses. Use inputmode to show the best keyboard on mobile devices.

    <input type="text" autocomplete="family-name">
<input type="email" autocomplete="email" inputmode="email">
<input type="tel" autocomplete="tel" inputmode="tel">

    Avoid using type="number" unless moving the value up or down makes sense, like choosing a quantity.

    Best practice: Build a Better Mobile Input

WCAG gives you the basics, but great accessibility comes from designing for the whole Accessibility Experience (AX). Aim to create components and pages that feel comfortable and intuitive for everyone to use.
